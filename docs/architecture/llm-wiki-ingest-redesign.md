# LLM Wiki Ingest Redesign

## North Star

Snowiki should become a Markdown-first, agent-operated LLM wiki substrate.

The core runtime remains CLI-first and deterministic. Claude, OpenCode, and future agent skills should operate Snowiki through stable CLI contracts instead of redefining a separate runtime. The LLM layer may summarize, transform, and decide what to file, but Snowiki core should reliably ingest, rebuild, query, lint, and report state without requiring an LLM call.

Target shape:

```text
Markdown/raw sources
  -> snowiki ingest
normalized/provenance manifest
  -> snowiki rebuild
compiled/searchable wiki artifacts
  -> snowiki query / lint / status
  -> Claude/OpenCode skills and workflows
```

This keeps Snowiki aligned with Karpathy's LLM Wiki pattern: durable knowledge is compiled into a persistent Markdown wiki instead of being re-derived from raw chunks on every query.

## Current Snowiki Gap

Snowiki currently behaves more like a downstream compiler for proprietary session exports:

```text
Claude/OpenCode export
  -> adapter-specific normalization
normalized JSON records
  -> rebuild
compiled Markdown pages
  -> query
```

That architecture is useful, but it is not yet a first-class Markdown wiki ingest system. The shipped `snowiki ingest` command is still centered on `--source claude` and `--source opencode`; Markdown files and directories are not the primary source surface.

Key gaps:

- Markdown files/directories are not first-class ingest inputs.
- Identity is record/session oriented, not `source_root + relative_path` oriented.
- Normalized storage is time-bucketed and not a latest-only path/content-hash upsert contract.
- Frontmatter is primarily generated by the compiler rather than preserved and promoted from source documents.
- Rebuild/query freshness semantics are not explicitly tied to a source content manifest.
- Claude/OpenCode session ingestion is too close to the core CLI surface instead of living mostly in agent/skill transformation workflows.

## Target Ingest Contract

Phase 1 should establish this CLI contract:

```bash
snowiki ingest <file-or-directory> [--source-root PATH] [--rebuild] [--output json]
```

Concrete examples:

```bash
snowiki ingest README.md --output json
snowiki ingest docs/ --output json
snowiki ingest docs/ --source-root docs --rebuild --output json
```

Required behavior:

- Accept Markdown files and directories as first-class inputs.
- Treat Markdown ingest as the default meaning of `snowiki ingest PATH`; do not require `--source`.
- Directory ingest is recursive by default.
- Markdown discovery includes `.md` and `.markdown` files. Non-Markdown files inside directories are skipped without error.
- Discovery does not follow symlinks. Hidden directories and Snowiki/internal storage directories such as `.git`, `.venv`, `node_modules`, `raw`, `normalized`, `compiled`, and `index` are skipped.
- One Markdown file is one logical document unit.
- Document identity is `source_root + relative_path`.
- If `--source-root` is omitted, it defaults to the ingested directory for directory ingest and to the parent directory for single-file ingest.
- `source_root` is resolved to an absolute path and the input path must be inside it.
- Content hash controls unchanged/changed detection.
- Content hash is SHA-256 over the copied Markdown bytes.
- Re-ingest performs latest-only upsert for the same identity.
- Copy source content into Snowiki-controlled raw storage while retaining provenance back to the original path.
- Preserve source frontmatter wholesale.
- Promote only safe frontmatter fields, such as `title`, `summary`, `description`, `tags`, `aliases`, `status`, `source`, `date`, `projects`, `decisions`, `questions`, `related`, and `topics`.
- Do not let user frontmatter override runtime-reserved fields such as Snowiki record IDs, raw references, source type, storage provenance, or recorded timestamps.
- Default ingest does not rebuild.
- Ingest returns `rebuild_required: true` when compiled/search artifacts may be stale.
- `--rebuild` runs rebuild after successful ingest.

Example JSON shape:

```json
{
  "source_root": "/repo/docs",
  "documents_seen": 12,
  "documents_inserted": 2,
  "documents_updated": 3,
  "documents_unchanged": 7,
  "documents_stale": 0,
  "rebuild_required": true
}
```

The Phase 1 JSON result may include a per-document `documents[]` array for tests and callers, with each entry carrying `relative_path`, `record_id`, `content_hash`, `status`, and normalized/raw paths. The aggregate count fields above remain the stable minimum contract.

## Target Storage Concepts

Snowiki should keep separate responsibilities:

1. **Source/provenance layer**: copied Markdown snapshot, original Markdown path, content hash, frontmatter, ingest time, source root, relative path.
2. **Normalized document layer**: deterministic JSON representation suitable for rebuild/search.
3. **Compiled/wiki layer**: generated or projected Markdown artifacts used by query, MCP, export, and skills.
4. **Search/index layer**: lexical-first retrieval snapshot; semantic/vector/graph layers remain optional extensions.

The source document must be treated as authoritative. Compiled pages are derived artifacts and may be rebuilt.

### Source Storage Decision

Phase 1 uses **copy + provenance** storage:

- Copy the ingested Markdown bytes into Snowiki-controlled raw storage at ingest time.
- Store the original absolute path, `source_root`, `relative_path`, content hash, and frontmatter in provenance metadata.
- Normalize from the copied snapshot rather than from a future live read of the external source.
- Keep the external path metadata so later Obsidian/vault graph workflows can reason about where the document came from.

Raw snapshots reuse Snowiki's existing SHA-addressed raw storage shape: `raw/markdown/<sha-prefix>/<sha-rest>`. Source path metadata belongs in the normalized record/provenance payload, not in the content-addressed raw filename.

This preserves reproducibility when the original file moves or disappears while still retaining enough path context for future vault-aware workflows.

### Phase 1 Compiled Output Decision

Phase 1 emits **summary pages only** from Markdown document records. It does not generate concept pages, entity pages, topic pages, graph artifacts, vector indexes, review queues, or MCP write artifacts. Those layers remain later extensions after deterministic Markdown ingest is stable.

In Phase 1, a summary page is a deterministic projection of the Markdown document record, not an LLM-generated summary. The compiled page should expose the document title, source summary/description when present, body text, normalized path, source identity, and provenance. LLM summarization remains outside deterministic ingest.

### Phase 1 Normalized Markdown Record

Markdown records use `source_type: "markdown"` and `record_type: "document"`. They are latest-only records keyed by a deterministic ID derived from canonical `source_root + relative_path`, while legacy Claude/OpenCode records continue to use date-bucketed paths.

Minimum payload fields:

- `title`: frontmatter `title`, otherwise first Markdown heading, otherwise file stem.
- `summary`: frontmatter `summary` or `description`, otherwise an empty string.
- `text`: Markdown body without frontmatter.
- `frontmatter`: preserved source frontmatter as parsed key/value data.
- `promoted_frontmatter`: safe promoted fields only.
- `reserved_frontmatter`: reserved fields observed in source frontmatter but blocked from overriding runtime fields.
- `source_path`: original absolute source path.
- `source_root`: canonical absolute source root.
- `relative_path`: source path relative to `source_root`.
- `content_hash`: SHA-256 of copied Markdown bytes.
- `source_metadata`: deterministic ingest metadata such as extension and byte size.

"Promote" means safe frontmatter fields are copied into normalized payload fields and are eligible for compiled page frontmatter. Unpromoted source frontmatter remains preserved under `frontmatter`; reserved fields are preserved under `reserved_frontmatter` for auditability but must not overwrite Snowiki-managed fields.

### Phase 1 Error and Safety Policy

- Missing input path: fail the command with non-zero exit and JSON error when `--output json` is requested.
- Single-file input with a non-Markdown extension: fail with a clear error.
- Non-Markdown files inside a directory: skip.
- Symlinks: skip; do not follow file or directory symlinks.
- Input outside explicit `--source-root`: fail.
- Unreadable Markdown file: fail the ingest instead of silently producing partial state.
- Frontmatter parse limitations: preserve deterministic simple YAML-like frontmatter; malformed or unsupported frontmatter should not be allowed to override reserved fields. If parsing cannot proceed deterministically, fail that ingest with a clear error.

### Legacy Session Ingest Decision

The primary `snowiki ingest PATH` surface should stop accepting Claude/OpenCode exports as core write paths in Phase 1. Session exports should move above Snowiki core: a skill or workflow converts Claude/OpenCode sessions into Markdown notes, then runs `snowiki ingest <note>`.

Existing normalized Claude/OpenCode records should remain readable by `rebuild`, `query`, and compatibility tests where feasible. This is a read/rebuild compatibility goal only. Phase 1 removes adapter-specific session export write paths from `snowiki ingest`, including hidden `--source claude|opencode` compatibility bridges.

## Reference Patterns to Adopt

Adopt these patterns from surveyed LLM Wiki implementations:

- Hash/manifest-based idempotent ingest.
- Immutable raw/source inputs separate from generated wiki output.
- `index.md`, `log.md`, and `overview.md` as navigation/context artifacts.
- `sources[]` provenance in frontmatter for generated pages.
- Structural wikilink parsing for cleanup and linting instead of substring matching.
- Explicit review/rebuild workflows rather than silent page overwrites.
- Lexical/BM25-first search, with vector/graph features as opt-in layers.

Avoid these patterns in the first implementation pass:

- Making vector search the source of truth.
- Auto-running expensive LLM compilation inside deterministic core ingest.
- Letting generated pages overwrite user-authored source files.
- Mixing raw source capture, curated Markdown, generated pages, and index state in one namespace.
- Adding daemon, MCP writes, graph expansion, or deep research before the core ingest contract is stable.

## Migration Plan

This document is the running architecture ledger for Markdown-first ingest. Keep it synchronized as implementation decisions change:

- When a Phase 1 decision becomes concrete, move it from **Open Questions** into the relevant decision or deliverable section.
- When a requirement is intentionally deferred, record the target phase and the reason it is not part of Phase 1.
- When implementation discovers a new seam, test requirement, or compatibility constraint, add it here before or alongside the code change.
- Do not leave destructive behavior implied. Pruning, source deletion, compiled cleanup, and MCP writes must remain explicit roadmap items until implemented and tested.

Every phase should finish with a **phase-end consistency and cleanup pass** before PR:

1. Re-check spec -> plan -> implementation alignment.
2. Fix contract mismatches before treating the phase as complete.
3. Add missing acceptance tests for safety, help surface, compatibility, and error behavior.
4. Perform bounded maintainability cleanup where it improves the current phase boundary.
5. Remove compatibility bridges when tests/workflows can be migrated to the new contract without losing read compatibility.
6. Record any larger refactor or product decision as a deferred item in this ledger.
7. Re-run required verification after cleanup.

The cleanup pass is not a license for broad unrelated refactors. It should focus on readability, module boundaries, test clarity, hidden compatibility bridges, and reusable seams that the next phase will depend on.

### Now vs Later Ledger

Phase 1 includes:

- Markdown file/directory ingest as the primary `snowiki ingest PATH` behavior.
- `--source-root` identity control.
- Safe recursive Markdown discovery.
- Copy + provenance raw source storage.
- Latest-only normalized Markdown document upsert keyed by `source_root + relative_path`.
- Frontmatter preservation with safe field promotion and reserved field blocking.
- Summary-only compiled output for Markdown document records.
- `--rebuild` and JSON freshness reporting.
- Legacy normalized read/rebuild/query compatibility where feasible.

Phase 1 explicitly does **not** include:

- Automatic pruning of removed source files.
- Cascade cleanup of generated pages, indexes, or wikilinks.
- Concept/entity/topic/graph generation from Markdown documents.
- LLM summarization inside deterministic ingest.
- Vector indexes, review queues, daemon behavior, or MCP writes.
- Claude/OpenCode session export writes in the primary ingest CLI.
- Hidden Claude/OpenCode session export write bridges in `snowiki ingest`.

Deferred items must remain visible in Phase 2+ below instead of disappearing from the plan.

Deliverables:

- Implement safe recursive Markdown discovery.
- Add `--source-root` for stable identity when ingesting files from a larger vault or documentation tree.
- Implement `source_root + relative_path` identity.
- Implement content-hash latest-only upsert.
- Copy source Markdown into raw storage and record provenance to the original file.
- Preserve and safely promote frontmatter.

Verification:

- Markdown file ingest test.
- Recursive directory ingest test.
- Frontmatter preservation/promotion test.
- Hash unchanged/update test.
- Rebuild flag test.
- Legacy normalized rebuild compatibility test if retained.
- Path safety tests for symlinks, hidden/internal directories, and traversal-like paths.
- Summary slug/path length regression test for long Markdown titles or bodies.

Implementation order:

1. Add Markdown parsing/discovery helpers under `src/snowiki/markdown/`.
2. Extend raw/provenance storage to persist copied Markdown snapshots plus source path metadata.
3. Add latest-only Markdown document storage in `src/snowiki/storage/normalized.py` without breaking date-bucketed legacy records.
4. Refactor `src/snowiki/cli/commands/ingest.py` to make Markdown file/directory ingest the primary path and remove required `--source`.
5. Update compiler summary handling for Markdown document records and bounded summary slugs.
6. Add tests before broad documentation/README updates so the new CLI contract is executable.
7. Update README and skill-facing docs only after runtime behavior matches this contract.

- Establish generated `index.md`, `log.md`, and `overview.md` semantics.
- Define frontmatter conventions for Snowiki-managed pages.
- Add status/lint checks for stale compiled artifacts and missing source paths.

Concrete follow-up work:

- Define a stable JSON shape for stale reports, including `relative_path`, `source_path`, `record_id`, `content_hash`, `state`, and `recommended_action`.
- Decide whether `documents_stale` remains an ingest-only aggregate or becomes shared output from `status`/`lint`.
- Document Snowiki-generated page frontmatter separately from user-authored source frontmatter.
- Update README and skill-facing docs only after the shipped CLI behavior is the runtime truth.

### Phase 3: Explicit Prune and Cleanup

Deliverables:

- Add an explicit `--prune` flag or `snowiki prune` command for removed source files.
- Require clear JSON reporting before destructive cleanup.
- Remove or archive normalized Markdown records only when their source identity is explicitly pruned.
- Evaluate nashsu-style cascade cleanup for generated pages:
  - remove pruned identities from generated page `sources[]`,
  - delete single-source summary pages when safe,
  - update index/navigation artifacts,
  - clean dead wikilinks structurally rather than by substring matching.

Concrete follow-up work:

- Add a dry-run-first prune flow with JSON output before any destructive operation.
- Require explicit source identity selection for pruning; do not infer destructive cleanup from a missing live file alone.
- Preserve an audit trail for pruned normalized records, either by archive path or tombstone record.
- Add tests for multi-source generated pages before enabling cascade cleanup.
- Keep cascade cleanup separate from Phase 1 rebuild so deterministic ingest remains non-destructive.

### Phase 4: Agent and Skill Workflow

Deliverables:

- Update Claude/OpenCode skills so they treat the CLI as runtime truth.
- Provide natural-language workflows for:
  - ingest this file/directory,
  - summarize this session into Markdown and ingest it,
  - rebuild the wiki,
  - query the wiki,
  - file a durable answer back into Markdown,
  - lint/status before continuing work.
- Keep mutation CLI-mediated and reviewable.
- Replace direct Claude/OpenCode export ingest workflows with session-to-Markdown conversion workflows.
- Keep direct Claude/OpenCode export ingest out of the primary runtime; if needed later, implement it as explicit compatibility tooling outside `snowiki ingest PATH`.

Example workflow:

```text
User: File this session into the wiki.
Agent:
  1. Summarizes durable decisions into a Markdown note.
  2. Writes frontmatter with tags/source/session metadata.
  3. Runs `snowiki ingest <note> --output json`.
  4. Runs `snowiki rebuild --output json` if needed or requested.
  5. Runs `snowiki query` to verify recall.
```

Deliverables:

- Add MCP write support only after CLI write contracts are stable.

### Long-Term Engine Portability

Snowiki may move performance-sensitive engine components to Rust later. Phase 1 Markdown ingest should not depend on that migration, but it must preserve clean boundaries so a future Rust engine can consume the same durable artifacts.

Portability constraints:

- Treat CLI arguments, JSON output, normalized records, raw storage paths, and compiled Markdown artifacts as the stable boundary.
- Avoid making Python-only runtime objects or Click callback internals the durable contract.
- Keep normalized Markdown document records deterministic and language-agnostic: JSON primitives, stable field names, SHA-256 hashes, and POSIX-style relative paths.
- Keep engine-facing behavior testable through files and CLI JSON rather than in-process Python calls only.
- Defer Rust-specific implementation choices until after the Markdown ingest/storage/rebuild contracts stabilize.

This means Phase 1 ingest is compatible with a future Rust engine as long as it continues to write deterministic raw/normalized/compiled artifacts and keeps the CLI contract authoritative.
