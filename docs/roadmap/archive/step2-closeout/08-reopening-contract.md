# Sub-step H: Reopening Contract

## Purpose

Freeze the rules of the new bounded Step 2 reopening program before any family admission, benchmark-asset strengthening, or new tokenizer implementation work begins.

This note exists to answer one question:

> What exactly is being reopened, and under what rules is that reopening allowed to proceed?

## Starting truth

The current candidate set is already canonically closed:

- **Local Closeout Outcome**: `benchmark-only/no runtime promotion`
- **Promoted Tokenizer**: `[NONE]`
- **Step 4 Unblocked**: `[NO]`

That closed line is not being undone.
This reopening is a **new bounded evidence program**, not residual cleanup on the old line.

## Reopening hypothesis

The reopening program tests a single bounded idea:

> A small, representative tokenizer-family comparison on a strengthened judged set may identify one family that materially and stably outperforms `regex_v1` enough to justify a separate runtime-promotion package, or it may conclusively show that no stable winner exists for the current corpus and benchmark design.

## Terminal outcomes

The reopening program may end in only one of the following states:

1. **Stable winner recommendation**
   - one tokenizer family clears the frozen bar
   - output is a recommendation package only
   - runtime promotion is still a later separate step
2. **No stable winner**
   - no family clears the bar, or rankings remain unstable
   - Step 2 closes again as `benchmark-only/no runtime promotion`
3. **Blocked-with-artifact**
   - required dependency or benchmark-asset governance prevents the minimally necessary work
   - the program closes with a canonical explanation of the smallest approval needed

## Explicitly out of scope

This reopening program does **not** do any of the following directly:

- runtime promotion
- Step 4 runtime implementation
- open-ended tokenizer architecture churn
- unbounded candidate-family expansion
- semantic/hybrid retrieval work as a substitute for lexical proof

## What stays fixed

The reopening program must preserve these truths unless later evidence changes them canonically:

- `regex_v1` remains the control baseline
- Step 4 remains blocked until a later separate recommendation/promotion path says otherwise
- null result is a valid successful closeout
- current candidate-set closure remains historically correct even while the new program is active

## What changes now

What changes with this note is only the execution posture:

- Step 2 is no longer in passive closeout-only mode
- a new bounded reopening program is now active
- the next mandatory decision is to freeze candidate-family scope and benchmark maturity rules before implementation continues

## Acceptance criteria

- the note explicitly states that the old line remains closed
- the note defines the reopening hypothesis
- the note defines exactly three terminal outcomes
- the note explicitly forbids runtime promotion in this reopening program
- the note states that Step 4 remains blocked
